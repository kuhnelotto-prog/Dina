
Контекст проекта: торговый бот Дина

Ты работаешь с проектом торгового бота «Дина» для биржи Bitget (фьючерсы USDT-Margined). Вот всё что нужно знать:

Архитектура:
Бот разделён на два автономных модуля — Дина-Лонг и Дина-Шорт. Шорт с более жёсткими фильтрами. Управляет ими Оркестрант. Три таймфрейма. 10 монет. Плечо 3x.

Ключевые компоненты:
• main.py — точка входа
• bitget_executor.py — исполнение ордеров (сейчас перегружен, ~38KB)
• risk_manager.py — риск-менеджмент
• learning_engine.py — адаптивные веса сигналов
• telegram_bot.py — уведомления с приоритетами
• orchestrator.py — управление ботами

Что уже реализовано:
• Trailing stop и trailing profit (4 этапа на ATR) — изначальные фичи автора
• Reconciliation при рестарте (с проверкой SL и аварийным SL = 1.5×ATR)
• Retry-логика с экспоненциальной задержкой
• Cold start LearningEngine (DEFAULT_WEIGHTS, MIN_TRADES_TOTAL=50)
• ExecutionGuard (allowlist, rate limits)
• Dry-run режим
• EventBus, DataFeed, SignalBuilder

Что ещё не реализовано (приоритет):
1. SafetyGuard — независимый watchdog-процесс (критично)
2. Дина-Шорт — конфигурация для отдельного запуска
3. commission в TradeRecord (из ответа биржи, не расчётная)
4. _check_correlated_exposure в risk_manager (группы L1/DeFi/Meme/AI/Gaming + BTC-поправка 0.5)
5. Ночные уведомления с приоритетами (critical/high/normal/info + SILENT_HOURS)

Важные решения:
• Плечо по умолчанию 3x,в перспективе адаптивное до 10х.
• VPS: Contabo SG1 (Сингапур), берём после первой рабочей версии
• GitHub — основное хранилище кода
• Оператор keepalive: если 7 дней нет команды → плавно закрыть позиции
• Автономный режим: Telegram может быть недоступен, бот продолжает работать
• Логирование для немецкой налоговой (commission фактическая из ответа биржи)

Стиль работы:
• Работаем пошагово, по одному файлу
• Сначала закрываем баги, потом оптимизации
• Не усложнять без необходимости
• WebSockets, Scanner — это будущее, не сейчас

Начнём с SafetyGuard — реализуй его как независимый процесс-watchdog.
